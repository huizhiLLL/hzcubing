# 项目现状

更新时间：2026-05-06

## 项目定位

HZCubing 是面向魔方玩家的社区成绩系统，核心围绕用户、成绩、榜单、个人主页和活动项目展开。

当前项目采用前后端分离方式维护：

- 前端：`src/`，Vue 3 单页应用，负责页面、交互、状态管理和 API 调用。
- 后端：`server/`，Express + MongoDB API 服务，负责鉴权、用户、成绩、榜单和整活项目数据。
- 静态资源：`public/`。
- 项目文档：`docs/`，按 `project.md`、`architecture.md`、`roadmap.md` 三份文档维护。

## 当前可用能力

### 前端

已完成的主要页面与能力：

- 首页：项目入口、成绩与用户概览、项目展示。
- 登录注册页：支持登录、注册和登录后跳转。
- 榜单页：支持按项目查看单次和平均榜单。
- GR 页：展示当前各项目最佳纪录与近期破纪录信息。
- 选手总览页：展示社区选手列表与概览信息。
- 个人主页：展示资料、个人最佳、最近成绩和创建的整活项目。
- 成绩提交页：支持提交、编辑、删除个人成绩。
- 设置页：支持头像、昵称、WCA ID、简介等资料维护。
- 整活项目管理页：管理员可维护动态整活项目。

当前前端已经形成的共享层：

- `src/api/index.js`：统一 axios 实例、鉴权 token 注入和 API 封装。
- `src/stores/user.js`：用户登录态、当前用户信息和本地持久化。
- `src/stores/records.js`：成绩加载、缓存复用、榜单与个人最佳相关能力。
- `src/stores/events.js`：官方项目、趣味项目、动态整活项目的合并与查询。
- `src/stores/theme.js`：主题状态。
- `src/utils/recordRanking.js`：前端榜单聚合与排名映射逻辑。

### 后端

已完成的主要接口域：

- `/api/auth`：注册、登录、获取当前用户。
- `/api/users`：用户资料、用户列表、用户总览、用户详情、QQ 相关查询和绑定。
- `/api/records`：成绩创建、查询、更新、删除、个人最佳、全站最佳、GR 历程、近期破纪录。
- `/api/meme-events`：整活项目公开查询和管理员 CRUD。
- `/health`：服务健康检查。

当前后端已具备的基础设施：

- Express 应用入口：`server/index.js`。
- MongoDB 连接：`server/config/database.js`。
- JWT 鉴权中间件：`server/middleware/auth.js`。
- 统一错误处理中间件：`server/middleware/errorHandler.js`。
- Mongoose 模型：`User`、`Record`、`MemeEvent`、`Counter`。

## 当前重点

当前项目已具备可用 MVP 主链路，但工程成熟度仍需要继续补强。现阶段重点不是扩大功能范围，而是让已有主链路更稳定、更一致、更容易继续迭代。

优先关注：

- 前后端职责边界保持清晰，避免把业务规则散落在页面或入口文件中。
- 前端继续沉淀共享 store、工具函数和公共组件，减少页面内重复逻辑。
- 后端逐步拆出 service 层，避免路由文件继续承载过多业务聚合。
- 对成绩排名、GR 计算、用户概览、整活项目变更等高风险逻辑补测试。
- 对 QQ / 外部插件入口补安全边界和调用治理。

## 暂不做的内容

除非有明确任务，不优先推进：

- 大规模 UI 重做或重新设计整套视觉系统。
- 切换前端框架、状态管理方案或构建工具。
- 切换后端框架、数据库或核心鉴权方案。
- 引入重量级组件库、ORM、任务队列或复杂微服务架构。
- 直接在远端服务器长期改代码。

- `docs/project.md`：项目定位、当前状态、完成能力、当前重点、暂不做内容。
