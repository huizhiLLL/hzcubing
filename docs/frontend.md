# 前端现状与下一步方向

日期：2026-03-21

## 文档目的

这份文档是当前前端的唯一对齐文档，用于回答三件事：

1. 当前前端到底做到了什么
2. 哪些能力还没做完，或者实现和原设想已经发生偏移
3. 下一步最值得继续投入的优化方向是什么

此前几份阶段性 PRD、复盘和收口文档已经完成历史使命，后续不再分散维护，统一以本文为准。

---

## 当前总判断

当前前端已经完成了魔方社区 MVP 的主体骨架，核心链路已经可用：

- 首页可进入主要功能
- 排行榜可按项目和单次/平均查看最佳记录
- 登录、注册、资料设置（含头像修改）、个人主页、成绩提交与个人成绩管理已打通
- 整活项目管理页已具备管理员 CRUD 能力
- `records`、`events` 两类共享数据和排名逻辑已经从页面内抽离出来

但当前状态还不能说“完全收尾”，主要原因有四个：

- 原设想中的“纪录时间轴 / 历史突破页”尚未真正实现
- 页面系统没有完全覆盖全站，部分页面还在旧结构上运行
- 用户总览仍偏页面内聚合，尚未形成统一共享层
- 前端几乎没有测试与回归保护

基于现有代码结构，当前前端可按两个口径判断：

- MVP 功能完成度：约 80%
- 工程化与一致性完成度：约 65%

这个判断是基于当前代码、路由、store、组件使用情况和生产构建结果得出的工程评估，不是产品承诺值。

## 当前前端结构

### 1. 应用壳层

入口和壳层已经稳定：

- `src/App.vue`
- `src/components/layout/MainLayout.vue`

当前具备：

- 悬浮式侧边栏布局
- 桌面端折叠态
- 移动端抽屉式导航
- 明暗主题切换
- 登录态相关入口与管理员入口

这部分已经不是主要风险点，后续以稳定维护为主。

### 2. 页面级公共层

当前已抽出的页面级基础组件：

- `AppPageHeader`
- `AppSectionCard`
- `AppFormActions`
- `AppStatusBlock`
- `AppSelect`

接入情况：

- 完整接入：`AuthView`、`SubmitRecordView`、`SettingsView`、`LeaderboardView`、`AdminMemeEventsView`
- 部分接入：`UserProfileView`、`ProfileHomeView`
- 未接入或仍主要使用旧页面结构：`HomeView`、`GRView`、`RecordHistoryView`

结论：

- 页面系统已经成立
- 但还没有做到“新增页面默认都在这套骨架里”
- 当前最容易继续分叉的页面正是 `GRView` 和 `RecordHistoryView`

### 3. 状态与数据层

当前共享状态主要集中在 4 个 store：

- `src/stores/user.js`
- `src/stores/records.js`
- `src/stores/events.js`
- `src/stores/theme.js`

当前真正完成共享收口的能力：

- 用户鉴权、本地 token 持久化、启动时登录态恢复
- `records` 的统一加载、排行榜聚合、个人最佳基础能力
- `events` 的统一加载、静态项目 + 动态整活项目合并
- 主题状态统一管理

当前仍偏页面内处理的能力：

- 选手总览的数据视图模型
- 部分页面的错误展示和降级逻辑
- 首页的异常状态反馈

### 4. 共享业务逻辑

当前已进入共享模块的关键逻辑：

- `src/utils/recordRanking.js`

已经收口的内容：

- 榜单最佳成绩聚合
- 各项目名次映射
- 个人最佳按全局名次排序

这块已经是当前前端最重要的共享业务层之一，后续排名规则调整应继续收在这里，不应再分散回页面。

## 当前完成度拆解

### A. 核心业务链路

完成度：高

已完成：

- 登录/注册
- 成绩提交
- 排行榜查看
- 个人主页查看
- 个人设置修改（含头像）

当前不足：

- 提交页还未提供 DNF / DNS 快捷输入能力
- 已支持在提交页右上角直接管理个人已提交成绩，但提交相关交互还缺少更细的快捷操作
- 首页没有承接“最近纪录突破”或“最新动态”模块

### B. 页面与导航层

完成度：中高

已完成：

- 主要导航稳定
- 路由守卫可用
- 管理员页权限入口存在

当前不足：

- 路由命名、文件命名和页面职责不完全一致
- 真正的“记录历程”页面缺位
- 个别页面仍游离于统一页面系统之外

### C. 数据共享与可维护性

完成度：中高

已完成：

- `records` 和 `events` 已形成共享加载入口
- 首页、榜单、个人主页已开始消费共享聚合结果
- 排名逻辑已共享

当前不足：

- `users` 侧还没有形成与 `records/events` 同等级的共享加载与视图层
- 页面级错误处理还不统一
- 401 处理仍采用整页刷新，比较粗糙

### D. 视觉统一与组件复用

完成度：中

已完成：

- 应用壳层风格统一
- 表单与状态块的统一明显改善
- 主要业务页视觉语言趋于一致

当前不足：

- 首页、GR、选手总览仍未完全并入页面系统
- 仍有闲置旧组件未清理

当前未被引用的公共组件：

- `src/components/common/UserCard.vue`
- `src/components/common/TimeDisplay.vue`
- `src/components/common/EventBadge.vue`

这些组件目前不应再被文档描述为“当前正在复用的核心公共组件”。

### E. 工程质量与回归保护

完成度：低

现状：

- `npm run build` 可通过
- 没有看到前端测试文件
- 共享排名逻辑和 store 缺少自动化回归保护

这部分是当前最明显的短板。

## 当前最关键的缺口

### 1. 真实的“纪录时间轴”还没做

这是与原前端设想偏差最大的一点。

当前：

- `/gr` 是当前各项目最佳纪录汇总
- `/players` 是选手总览

但缺少：

- 按项目查看纪录被谁、在什么时候打破
- 纪录演进的时间轴或列表

因此当前文档不能再把“记录历程页已完成”写成既成事实。

### 2. 页面系统还没有全站收口

当前公共页面骨架已经很好用，但 `HomeView`、`GRView`、`RecordHistoryView` 仍没有充分纳入。

这会带来两个问题：

- 视觉和结构容易再次分叉
- 后续继续扩页时难以保持统一节奏

### 3. 用户总览还没有沉淀成共享能力

`/players` 页面仍然自己调用 `userAPI.getOverview(...)` 并在页面中持有分页和展示逻辑。

这本身不是 bug，但说明“用户侧共享层”还没有真正建立。

### 4. 质量保护明显不足

目前共享逻辑越多，越需要少量但关键的测试保护。

当前最值得优先补的不是大而全的测试体系，而是：

- `recordRanking.js` 的排名边界测试
- `recordsStore` / `eventsStore` 的加载复用测试
- 关键页面最基础的 smoke test

## 下一步优化方向

### 第一优先级

1. 明确 `GRView`、`RecordHistoryView`、`/players` 的职责边界  
   要么补出真正的纪录时间轴页，要么重命名页面与路由，先把概念对齐。

2. 继续把剩余页面接入统一页面骨架  
   优先处理 `GRView` 和 `RecordHistoryView`，其次看 `HomeView` 是否需要补轻量级的统一题头或状态块。

3. 建立 `users` 侧共享层  
   至少把选手总览所需的加载、分页和摘要能力抽到 store 或 composable。

4. 给共享逻辑补最小测试集  
   先保住排名逻辑和加载策略，再考虑扩展。

### 第二优先级

1. 优化成绩提交体验  
   包括 DNF / DNS、提交后去向、输入容错、移动端键盘体验。

2. 完善首页的信息密度  
   补充最近动态、GR 亮点或近期活跃内容，让首页不只承担导航作用。

3. 清理闲置组件与命名遗留  
   处理不再使用的组件，以及 `RecordHistoryView.vue` 这类已与实际职责脱节的命名。

### 第三优先级

1. 统一错误反馈和加载反馈
2. 调整 401 处理策略，减少整页刷新
3. 视实际需要再补组件级抽象，而不是提前扩张组件库

后续前端文档建议只保留这一个入口文档，并遵循下面的规则：

- 重大页面职责变化，直接更新本文件的“信息架构”和“完成度拆解”
- 新增共享层能力，直接更新本文件的“结构”与“下一步优化方向”
- 阶段性复盘不再单独新增同类文档，避免重复和过期

## 一句话结论

当前前端已经完成了可用的社区 MVP 主链路，最值得继续投入的不是再做一轮零散视觉优化，而是把页面职责、统一骨架、用户侧共享层和测试保护补齐，让这套前端从“能用”进一步进入“稳定、可维护、可持续演进”的状态。
