# 路线图

更新时间：2026-05-06

## 当前阶段目标

当前阶段目标是把已经可用的魔方社区 MVP 主链路稳定下来，而不是快速扩张功能范围。

核心判断：

- 前端主页面和主要交互已经可用。
- 后端用户、成绩、榜单和整活项目接口已经可用。
- 当前短板主要在测试、接口契约、性能边界、安全边界和模块分层。

## 近期优先级

### P0：稳定主链路

- 补充成绩排名、个人最佳、GR 计算的最小测试集。
- 补充注册登录、成绩增删改查、整活项目管理的接口回归测试。
- 对齐 `server/package.json` 中脚本与真实仓库文件，避免保留不可执行脚本入口。
- 统一关键页面的加载、错误和空状态反馈。

### P1：前端收口

- 继续将页面接入统一页面骨架和公共组件。
- 将选手总览相关加载、分页和视图派生能力沉淀到共享层。
- 清理未使用或职责过期的公共组件。
- 优化成绩提交体验，包括 DNF / DNS、输入容错和移动端输入体验。
- 梳理 GR 页职责，明确“当前纪录看板”和“纪录时间轴”的边界。

### P1：后端收口

- 优先优化 `/api/records/best`、`/api/records/recent-breaks` 等聚合接口。
- 为 QQ / AstrBot 外部入口补请求来源识别、简单签名、速率限制和结构化日志。
- 抽出 `recordService`、`userService`、`authService` 等服务层雏形。
- 统一分页响应、成功状态码和用户标识字段。

## 中期计划

- 建立最小 API 契约文档或 OpenAPI 描述。
- 明确官方项目、趣味项目、整活项目的事件字典关系。
- 为管理员补用户状态管理、角色调整、成绩修正等治理能力。
- 为关键写操作补审计日志。
- 根据数据量评估排行榜缓存、预计算或物化统计。

## 暂缓事项

以下事项暂缓，除非出现明确需求：

- 大规模重做 UI。
- 多端客户端开发。
- 微服务化。
- 更换数据库。
- 引入复杂权限系统。
- 引入任务队列、搜索服务或 BI 分析系统。
- 完整比赛系统、报名系统、赛程系统。

## 技术决策记录

### 前后端分离

继续保持 `src/` 前端和 `server/` 后端分离。前端只通过 HTTP API 访问数据，后端承接可信校验和数据库读写。

### 文档结构

后续只维护三份主文档：

- `project.md`
- `architecture.md`
- `roadmap.md`

不再按前端、后端分别维护长期状态文档，避免进度、架构和计划分散重复。

### 改造方式

后续改造遵循小步推进：

- 先补测试和契约。
- 再优化性能和安全边界。
- 最后做结构分层和治理能力扩展。

不为了结构完整性进行一次性大重构。
