# 后端现状与改进方向

日期：2026-03-22

## 文档目的

这份文档用于统一回答三件事：

1. 当前 `server/` 后端到底已经做到了什么
2. 现在的完成度更接近“功能可用”还是“工程成熟”
3. 下一步最值得投入的改进方向是什么

后续涉及后端较大的功能补齐、结构调整或能力下沉时，统一更新本文，不再新增同类评估文档分散维护。

---

## 当前总判断

当前后端已经完成了魔方社区 MVP 所需的核心数据服务骨架：

- 用户注册、登录、鉴权和当前用户信息获取已打通
- 用户资料修改、用户列表、用户总览、用户详情接口已具备
- 成绩提交、修改、删除、分页查询、个人最佳、全站最佳、最近破纪录列表已具备
- 整活项目（`meme-events`）已有管理员 CRUD
- MongoDB 模型、基础索引、统一错误处理中间件已经建立

但它仍明显停留在“可用的单体后端 MVP”，离“稳定、可扩展、可回归”的状态还有距离，主要短板集中在：

- 缺少自动化测试、契约校验和回归保护
- 路由层承担了过多聚合和业务计算，缺少 service 层
- 若数据量继续增长，部分接口的查询方式会出现性能压力
- 外部 QQ / AstrBot 接口已经接入，但安全边界和治理还比较粗
- `package.json` 声明了多条迁移/测试脚本，但仓库内当前并没有对应 `server/scripts/` 目录，运维与数据维护能力未真正闭环

基于现有代码，我对后端的整体判断是：

- MVP 功能完成度：约 78%
- 工程化与可维护性完成度：约 55%

这是一份工程评估值，不是产品承诺值。

---

## 当前实际能力边界

### 1. 应用入口与基础设施

后端入口集中在 `server/index.js`，当前已具备：

- Express 应用初始化
- CORS 白名单配置
- `morgan` 日志
- JSON / 表单解析
- `/health` 健康检查
- 路由注册
- 统一错误处理中间件

已落地的路由前缀只有四组：

- `/api/auth`
- `/api/users`
- `/api/records`
- `/api/meme-events`

说明当前后端边界很清晰，尚未扩展成多域后台，也还没有后台任务、文件服务、消息服务之类的配套模块。

### 2. 用户与鉴权

当前用户侧已完成：

- 邮箱注册、登录、JWT 鉴权
- 当前登录用户信息获取
- 用户资料更新
- 用户列表分页
- 用户总览卡片数据
- 用户详情查询
- `userNo` 自增编号生成

同时还包含一条偏业务化的外部绑定链路：

- QQ ID 绑定
- 通过昵称绑定 QQ
- 通过 QQ 查询用户

这说明“用户身份 + 社区账号 + 外部插件接入”这条链路已经开了口子，但治理还不够严谨。

### 3. 成绩与榜单

这是当前后端完成度最高的部分。

已提供：

- 成绩创建、修改、删除
- 全站成绩分页
- 用户成绩分页
- 全站各项目最佳成绩
- 用户各项目最佳成绩
- 用户成绩历史
- 最近破纪录列表

此外，成绩模型已经包含：

- 单次、平均
- 器材 `cube`
- 复原方式 `method`
- 比赛名 `competition`
- 原始 solves 数组
- 时间格式化虚拟字段

说明记录与展示成绩这条主链路已经被后端真实承接起来了，不再只是表面接口。

### 4. 整活项目

当前已支持：

- 公开获取整活项目列表
- 管理员新增、编辑、启停、删除整活项目
- 修改整活项目 `eventCode` 时同步迁移该项目下的成绩记录
- 删除整活项目时同步删除该项目下的成绩记录，避免遗留孤儿数据

这块与前端的管理员页是对齐的，属于“功能已可用，但规模还不大”的状态。

## 当前结构判断

### 1. 路由层偏重，服务层缺失

目前业务逻辑几乎都堆在 `routes/*.js` 里，模型层只负责 schema，本质上还是：

- route 里做校验
- route 里查库
- route 里做聚合
- route 里拼响应

这种写法在 MVP 阶段很高效，但继续演进会带来三个问题：

1. 逻辑复用困难  
   比如成绩最佳值、破纪录识别、用户聚合统计都没有沉到共享服务。

2. 测试粒度很粗  
   现在几乎只能测接口，很难对核心规则做单元测试。

3. 路由文件会越来越长  
   `records.js` 已经明显承担了过多职责。

### 2. 数据模型基础可用，但领域约束还不够完整

`User`、`Record`、`MemeEvent`、`Counter` 这四个模型足够支撑 MVP。

当前做得比较好的地方：

- `User` 密码哈希、唯一邮箱、唯一 `qqId`、`userNo` 自增都已实现
- `Record` 已对 `userId`、`event`、`timestamp` 和成绩字段建立索引
- 时间字段做了两位小数截断

但还存在几个明显的领域空白：

- `event` 仍是自由字符串，没有统一事件字典或枚举约束
- `Record.nickname` 与 `User.nickname` 是冗余字段，当前策略偏“写时快照”，但文档和约束没有写清
- `competition`、`solves` 已建模，但目前主要接口几乎没有真正消费它们
- 没有软删除、审计日志、操作记录之类的治理能力

### 3. API 返回风格基本统一，但还不算稳定契约

当前大部分接口都遵循：

- `code`
- `message`
- `data`

有些分页接口还返回：

- `page`
- `pageSize`
- `total`

这个方向是对的，但离“清晰 API 契约”还差一步，因为：

- 有的接口 `201` 状态码下仍返回 `code: 200`
- 外部接口和内部接口的字段命名不完全统一
- 用户标识有时返回 `_id`，有时主推 `userNo`，有时又叫 `profileUserNo`

当前更像“整体风格接近统一”，而不是“严格受约束的接口规范”。

## 完成度拆解

### A. 核心业务链路

完成度：高

已完成：

- 注册 / 登录 / 获取当前用户
- 个人资料更新
- 成绩提交、编辑、删除
- 榜单与个人最佳查询
- 最近破纪录数据
- 整活项目管理员维护

当前不足：

- 缺少真正的管理员用户管理、封禁、角色调整接口
- 没有成绩审核机制
- 没有历史纪录修正、批量导入、批量回滚能力

### B. 外部插件接入

完成度：中

已完成：

- QQ 绑定
- 通过 QQ 查找用户
- 通过 QQ 直接提交成绩

当前不足：

- 对外接口基本是公开的，没有签名、速率限制或来源校验
- 通过昵称绑定 QQ 依赖昵称唯一语义，但模型层并未强制昵称唯一
- 外部入口与站内标准鉴权链路是并行分叉的，后续容易失控

### C. 数据查询与聚合

完成度：中高

已完成：

- 用户总览聚合统计
- 全站最佳成绩计算
- 最近破纪录计算

当前不足：

- 多个接口仍采用“先全量查出，再在 Node 里遍历聚合”的方式
- 数据量增大后，`/api/records/best`、`/api/records/recent-breaks` 会最先成为性能瓶颈
- 用户成绩历史和用户成绩分页接口职责几乎重复，还没有真正区分“历史视图”和“列表视图”

### D. 安全与边界控制

完成度：中低

已完成：

- JWT 鉴权
- 基础角色判断
- 基础参数校验
- 统一错误响应
- CORS 白名单

当前不足：

- 没有 rate limit、防暴力破解、防刷提交
- 没有更细粒度的输入约束和字段白名单
- 没有审计日志和敏感行为记录
- 对外公开的 QQ 相关接口缺少防滥用保护

### E. 工程质量与运维准备

完成度：低

现状：

- 没有发现测试文件
- `package.json` 中声明了多条迁移/测试脚本，但仓库当前没有对应 `server/scripts/` 目录
- 没有 service / repository 分层
- 没有 OpenAPI / 接口文档 / schema 契约

这部分是当前后端最薄弱的区域。

## 当前最关键的问题

### 1. 性能风险已经埋下，但还没有治理

目前多个查询接口直接把整批记录拉到内存里再做计算，例如：

- `/api/records/best`
- `/api/records/user/:userId/best`
- `/api/records/recent-breaks`

在当前数据量下可能还够用，但只要社区活跃增长，这类写法就会让后端先在排行榜和纪录页上吃到压力。

### 2. 外部接口已上线，但安全设计不足

QQ/AstrBot 相关接口说明后端已经开始被站外系统消费，这是好事，但风险也更高。

当前最需要补上的不是新功能，而是：

- 请求来源识别
- 速率限制
- 最小暴露字段
- 错误与操作日志

否则一旦被误调或滥用，最先出问题的就是绑定和成绩提交。

### 3. 路由重复和职责重叠开始出现

当前后端已经出现一些信号：

- `records` 路由体量过大
- `history` 与 `user records` 返回结构高度相似
- 路由里已有未被真正接入的辅助函数/设计痕迹

这说明它还没乱，但已经到了应该开始收口的时候。

### 4. 运维与数据维护链路没有真正闭环

`package.json` 里已经声明了迁移、补数据、测试 API 等脚本入口，说明项目曾经有过这方面需求，但当前仓库内看不到相应目录。

这会带来两个现实问题：

1. 新协作者会误以为这些维护能力可用  
2. 真到需要迁移、修数据、回放时，缺少可执行资产

这个问题不影响日常开发，但会在数据演进阶段集中暴露。

## 下一步改进方向

### 第一优先级

1. 给成绩相关接口做一次性能收口  
   优先处理 `/api/records/best` 和 `/api/records/recent-breaks`，尽量改为 Mongo 聚合或增量预计算思路。

2. 给 QQ / AstrBot 外部接口补安全边界  
   至少补上简单签名、来源约束、速率限制和结构化日志。

3. 建立最小测试集  
   先覆盖鉴权、注册登录、成绩增删改查、最佳成绩和最近破纪录这几条主链路。

4. 对齐脚本与实际仓库  
   要么补回 `server/scripts/`，要么清理失效 npm scripts，避免“看起来能跑、其实不存在”。

### 第二优先级

1. 拆出 service 层  
   建议优先抽出 `authService`、`recordService`、`userOverviewService`，把聚合逻辑从路由中移走。

2. 统一接口契约  
   明确用户主标识到底以 `userNo` 还是 `_id` 为准，分页返回统一结构，成功状态码与 `code` 字段统一。

3. 补管理员治理能力  
   包括用户状态管理、角色调整、成绩管理或成绩修正能力。

4. 明确事件体系  
   把官方项目与整活项目的事件定义关系补清楚，减少 `event` 自由字符串长期扩散。

### 第三优先级

1. 增加 OpenAPI 或最小接口说明文档
2. 为关键写操作补审计日志
3. 视实际需要再考虑缓存、排行榜快照或物化统计

---

## 建议的演进顺序

如果只允许做一轮务实改造，建议按下面顺序推进：

1. 先补测试和脚本对齐，保住基本可维护性
2. 再优化成绩聚合接口，保住性能上限
3. 然后收口 QQ 外部接口的安全边界
4. 最后再做 service 分层和管理能力扩展

这样做的好处是：

- 不会一下子把后端推成大重构
- 能优先解决真实风险，而不是只做结构美化
- 与当前项目“小步、可验证、可回退”的协作方式一致

---

## 一句话结论

当前后端已经把魔方社区最关键的“用户 + 成绩 + 榜单 + 活动项目”主链路撑起来了，属于一个功能可用的 MVP 服务端；接下来最值得投入的不是继续堆接口，而是补测试、收性能、加安全边界、理顺脚本和分层，让它从“能支撑当前页面”进入“能稳定承接后续迭代”的状态。
