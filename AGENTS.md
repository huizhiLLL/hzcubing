# AGENTS.md

## 项目定位

- 这是一个魔方社区系统，核心是用户、成绩、榜单、个人主页与活动项目。
- 项目采用前后端分离：`src/` 是 Vue 3 前端，`server/` 是 Express + MongoDB 后端。
- 默认用中文进行开发沟通、分析、计划、变更说明、代码注释和文档更新说明。

## 理解顺序

开始功能开发、Bug 修复、重构或较大改动前，先按顺序了解项目：

1. 当前任务描述。
2. 当前代码实现。
3. `docs/` 下的相关文档。
4. `README.md`。
5. `ref/` 下参考资料，仅作为辅助参考。

如果文档和代码不一致，以代码作为当前真实状态；架构规则和长期方向结合 `docs/architecture.md` 判断。

## 文档职责

项目文档统一维护在三份 Markdown 中：

- `docs/project.md`：项目定位、当前状态、当前重点、暂不做内容。
- `docs/architecture.md`：技术栈、前后端边界、模块职责、接口和数据约束。
- `docs/roadmap.md`：后续计划、优先级、暂缓事项、重要技术决策。

完成以下改动后必须检查是否同步文档：

- 较大功能开发。
- 重要模块修改。
- 架构、接口、协议、数据结构变化。
- 目录结构变化。
- 影响使用方式、构建方式或运行方式的变化。

README 只写项目介绍、功能、安装、运行和使用方式，不写临时进度和短期 TODO。

## 协作原则

- 先理解现状，再修改代码；优先基于现有结构演进，不轻易推翻重写。
- 改动保持小步、可验证、可回退；一次只解决一个明确问题。
- 默认做最小必要改动，避免无关重构、无关格式化、无关重命名。
- 遇到需求与现有实现冲突时，先写清取舍，再动手修改。
- 不确定时说明缺少的信息，提出需要确认的问题，不硬猜。

## 前端约定

- 继续使用 Vue 3 + Pinia + Vue Router 现有模式。
- 前端只通过 `src/api/index.js` 调用后端接口，不直接访问数据库。
- 优先复用 `src/components`、`src/stores`、`src/utils` 和现有样式变量。
- 避免把可复用业务逻辑堆在页面组件里。
- 涉及页面结构或视觉调整时，同时检查桌面端与移动端表现。
- 不新增重复组件、重复视觉规则或不必要的大型 UI 依赖。

## 后端约定

- 继续使用 Express + MongoDB + Mongoose 现有模式。
- API 返回结构、鉴权方式、数据模型命名保持一致，除非有明确迁移方案。
- 后端负责可信校验、权限判断、数据库读写和核心业务规则。
- 路由层避免继续膨胀；复杂业务优先下沉到 service 或共享工具模块。
- 涉及整活项目 `eventCode`、用户标识、成绩数据结构等持久化语义时，必须先评估数据影响。

## 风险操作

执行以下操作前，必须先说明计划、影响范围、潜在风险，并等待确认：

- 删除大量文件。
- 执行可能破坏数据的命令或脚本。
- 修改数据库迁移、持久化数据结构或用户数据。
- 推送代码。
- 发布、部署、覆盖线上环境。
- 引入大型依赖或切换核心技术方案。

## 修改边界

- 不随意改 `.env`、部署配置、数据库字段含义。
- 不引入重量级依赖，除非现有方案无法满足需求且收益明确。
- 前后端代码改动默认只在本地仓库完成；远端服务器仅用于临时脚本执行、数据库操作与运行状态排查。
- 远程服务器连接参数如需使用，以 `.env` 为准，但不要把敏感信息写入文档或提交内容。
